# プライバシー設計

## 外部通信が発生する場面

このプロジェクトは完全に外部通信ゼロではない．主に次の場面で外部通信が発生する．

```text
・初回の Docker image 取得
・Pi-hole のブロックリスト更新
・Unbound によるDNS再帰問い合わせ
・Uptime Kuma で外部URLを監視した場合
・NetAlertX や Uptime Kuma で通知連携を設定した場合
```

## 外部に送信されにくい情報

初期状態では，自宅 LAN の端末一覧や DNS ログを外部サービスへ送信する設定にはしていない．ただし，通知連携や外部連携を後から設定した場合は，IP アドレスや端末名などが通知先へ送信される可能性がある．

## Unbound の役割

Pi-hole の上流に Unbound を置くことで，Google DNS や Cloudflare DNS などの外部 DNS 事業者に問い合わせ履歴が集中する構成を避けられる．ただし，DNS の仕組み上，ルート DNS，TLD DNS，権威 DNS などへの問い合わせは必要に応じて発生する．
